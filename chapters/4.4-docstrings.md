# 4.4 Docstrings — Human Language Inside the Machine

## Why Docstrings Exist

Code tells you **what** a function does. A docstring tells you **why it exists and what to expect.**

There is a difference between knowing how something works and knowing what it is for. The code handles the first. The docstring handles the second.

```python
# Code without docstring — you can infer, but must read carefully
def process(data, threshold=0.5):
    return [x for x in data if x >= threshold]

# Code with docstring — intent immediately clear
def filter_above_threshold(data: list, threshold: float = 0.5) -> list:
    """
    Return elements from data that are greater than or equal to threshold.

    Args:
        data: List of numeric values to filter.
        threshold: Minimum value to include. Defaults to 0.5.

    Returns:
        New list containing only values >= threshold.

    Examples:
        >>> filter_above_threshold([0.3, 0.5, 0.7, 0.9], 0.5)
        [0.5, 0.7, 0.9]
    """
    return [x for x in data if x >= threshold]
```

## Docstring Formats

Three standard formats. Choose one and use it consistently:

### Google Style (recommended for most projects)

```python
def calculate_batting_average(hits: int, at_bats: int) -> float:
    """
    Calculate a player's batting average.

    Args:
        hits: Number of successful hits.
        at_bats: Total number of at-bats.

    Returns:
        Batting average as a float between 0.0 and 1.0.
        Returns 0.0 if at_bats is zero.

    Raises:
        ValueError: If hits or at_bats are negative.

    Examples:
        >>> calculate_batting_average(34, 100)
        0.34
    """
```

### NumPy Style (for scientific/data code)

```python
def normalize(values: list) -> list:
    """
    Normalize a list of values to the range [0, 1].

    Parameters
    ----------
    values : list
        List of numeric values to normalize.

    Returns
    -------
    list
        Values scaled to [0, 1] range.
    """
```

### reStructuredText Style (Sphinx/Python stdlib)

```python
def greet(name: str) -> str:
    """
    Generate a greeting for the given name.

    :param name: The name to greet.
    :type name: str
    :returns: Formatted greeting string.
    :rtype: str
    """
```

## Accessing Docstrings

```python
print(calculate_batting_average.__doc__)   # Print the docstring
help(calculate_batting_average)            # Full formatted help
```

Good tools (IDEs, documentation generators like Sphinx) parse docstrings to build documentation automatically. Write them for humans — machines will handle the rest.

---

## What a Docstring Should NOT Do

A docstring should explain **purpose and contract** — not implementation:

```python
# Bad docstring — describes implementation (the code already does this)
def add(a, b):
    """Add a to b using the + operator and return the result."""
    return a + b

# Good docstring — describes purpose and contract
def add(a: float, b: float) -> float:
    """Return the arithmetic sum of two numbers."""
    return a + b
```

If you find yourself writing a docstring that just restates the code, your code is already clear enough. If you find yourself writing a complex docstring to explain confusing code — refactor the code instead.

## One-Line Docstrings

For simple functions, one line is enough:

```python
def is_elite_hitter(avg: float) -> bool:
    """Return True if batting average qualifies as elite (>= 0.300)."""
    return avg >= 0.300
```

The entire docstring fits on one line. No Args, Returns, or Examples needed for something this clear.
