# 6.3 assert and fixtures — The Grammar of a Test

## `assert` — The Declaration of Expectation

At the heart of every test is an assertion: a statement of what you expect to be true.

```python
assert expression           # Pass if expression is truthy
assert expression, message  # Pass if truthy, show message on failure
```

`assert` is Python's most direct way to declare a belief:

```python
def test_batting_average():
    result = calculate_batting_average(3, 10)
    assert result == 0.3, f"Expected 0.3, got {result}"
```

If `result` is `0.3`, the test passes silently. If it is anything else, the test fails with your message.

---

## The Anatomy of a Good Assertion

pytest's failure messages are already excellent — they show you the actual vs expected values. But a custom message (`assert condition, "message"`) adds context when needed:

```python
# pytest will show: AssertionError: assert 0.25 == 0.3
def test_batting_average_basic():
    assert calculate_batting_average(3, 10) == pytest.approx(0.3)

# With custom message — more context in failure
def test_batting_average_edge():
    result = calculate_batting_average(0, 0)
    assert result == 0.0, f"Zero at-bats should return 0.0, got {result}"
```

## Testing Exceptions

Sometimes the correct behavior *is* raising an exception. Test for it explicitly:

```python
import pytest

def test_negative_at_bats_raises():
    with pytest.raises(ValueError):
        calculate_batting_average(3, -1)

# Inspect the exception message
def test_negative_hits_message():
    with pytest.raises(ValueError, match="negative"):
        calculate_batting_average(-1, 10)

# Capture the exception for deeper inspection
def test_invalid_average_custom_exception():
    with pytest.raises(InvalidAverageError) as exc_info:
        validate_batting_average(1.5)

    assert "1.5" in str(exc_info.value)
```

`pytest.raises` is the correct way to assert that code *should* raise. Without it, a test that expects an exception but doesn't get one will silently pass — which is the worst kind of test.

## Fixtures — Preparation as a Reusable Resource

A **fixture** is a function decorated with `@pytest.fixture` that provides setup (and optionally teardown) for tests.

Without fixtures, every test that needs a player dict would create it inline:

```python
# Without fixtures — repetitive
def test_get_hits():
    player = {"name": "Cosmos", "hits": 34, "at_bats": 100}
    assert player["hits"] == 34

def test_get_average():
    player = {"name": "Cosmos", "hits": 34, "at_bats": 100}
    assert player["hits"] / player["at_bats"] == pytest.approx(0.34)
```

With fixtures, the setup is written once and injected:

```python
import pytest

@pytest.fixture
def sample_player():
    """A standard player for testing stats functions."""
    return {
        "name": "Cosmos",
        "hits": 34,
        "at_bats": 100,
        "position": "pitcher"
    }

def test_get_hits(sample_player):
    assert sample_player["hits"] == 34

def test_get_average(sample_player):
    avg = sample_player["hits"] / sample_player["at_bats"]
    assert avg == pytest.approx(0.34)
```

pytest sees the `sample_player` parameter name, finds the fixture with the same name, calls it, and injects the result.

## Fixture Scope

By default, a fixture runs fresh for each test. You can control this:

```python
@pytest.fixture(scope="function")   # Default — once per test
def fresh_player():
    return {"name": "Cosmos", "hits": 0}

@pytest.fixture(scope="module")     # Once per file
def database_connection():
    conn = connect_to_db()
    yield conn
    conn.close()    # Cleanup after yield

@pytest.fixture(scope="session")    # Once per entire test run
def shared_config():
    return load_config()
```

Use `yield` in fixtures when you need teardown code — cleanup happens after `yield`.

## Fixtures for Objects with State

Fixtures are essential for testing stateful objects:

```python
class Stack:
    def __init__(self):
        self._items = []

    def push(self, item):
        self._items.append(item)

    def pop(self):
        if not self._items:
            raise IndexError("Stack is empty")
        return self._items.pop()

    def peek(self):
        if not self._items:
            raise IndexError("Stack is empty")
        return self._items[-1]

    def is_empty(self) -> bool:
        return len(self._items) == 0


# Fixtures for different initial states
@pytest.fixture
def empty_stack():
    return Stack()

@pytest.fixture
def populated_stack():
    stack = Stack()
    stack.push(1)
    stack.push(2)
    stack.push(3)
    return stack


# Tests use exactly the state they need
def test_push_to_empty(empty_stack):
    empty_stack.push(42)
    assert empty_stack.peek() == 42

def test_pop_from_populated(populated_stack):
    assert populated_stack.pop() == 3

def test_pop_from_empty_raises(empty_stack):
    with pytest.raises(IndexError):
        empty_stack.pop()
```

Each test gets its own instance. No state bleeds between tests.

## The Structure of a Trustworthy Test

A good test follows the **Arrange, Act, Assert** pattern:

```python
def test_calculate_batting_average(sample_player):
    # Arrange — set up the context
    hits = sample_player["hits"]
    at_bats = sample_player["at_bats"]

    # Act — execute the thing being tested
    result = calculate_batting_average(hits, at_bats)

    # Assert — verify the outcome
    assert result == pytest.approx(0.34)
```

One test, one behavior, one assertion (or a small, related cluster). A test that checks ten unrelated things tells you *something failed* but not *what to fix.*

Precision in tests mirrors precision in thought.
