# 1.3 Version Control with Git and GitHub

## Why Git Exists

Code evolves. And evolution without memory is chaos.

Imagine writing a 500-line program, making a change, and breaking everything — with no way to go back. That is what programming felt like before version control.

**Git is the eternal memory of code.** It tracks every change you make, lets you revert to any previous state, enables collaboration, and lets you experiment without fear.

Every serious programmer uses Git. Not knowing Git is like a writer who doesn't know how to save their documents.

---

## Git vs GitHub

**Git** is the tool — software that runs on your machine and tracks changes locally.

**GitHub** is the platform — a cloud service where you store your Git repositories online, share them with others, and collaborate.

Git can exist without GitHub. GitHub cannot exist without Git.

---

## Essential Git Commands

```bash
# Initialize a repository in your current folder
git init

# Check the current state of your files
git status

# Stage all changes for commit
git add .

# Stage a specific file
git add main.py

# Save a snapshot with a message
git commit -m "Add batting average function"

# View the history of commits
git log --oneline

# Create a new branch
git checkout -b feature/new-function

# Switch branches
git checkout main
```

---

## The Git Workflow

Think of commits as **meditation checkpoints** — moments where you say *"this is where I am, and it is good."* Then you continue forward, knowing you can always return.

```
1. Write code
2. git status     → see what changed
3. git add .      → stage what you want to save
4. git commit -m  → create the checkpoint
5. Repeat
```

Good commit messages describe **why**, not what:

```bash
# Bad — describes what (the code already says what)
git commit -m "change variable name"

# Good — describes why
git commit -m "Rename x to batting_average for clarity"
```

---

## Connecting to GitHub

```bash
# Connect your local repo to GitHub (after creating repo on github.com)
git remote add origin https://github.com/yourusername/your-repo.git

# Push your commits to GitHub
git push -u origin main

# Pull changes from GitHub
git pull origin main
```

---

## The .gitignore File

Some files should never be committed — virtual environments, API keys, compiled files:

```gitignore
# .gitignore
.venv/
__pycache__/
*.pyc
.env
```

Create this file at the root of every project. Put it before your first commit.

---

## The Philosophy

Git teaches something profound: **nothing is permanent, and everything is recoverable.**

You are free to experiment because you can always go back. You are free to collaborate because changes are tracked. You are free to break things because the history preserves what worked.

That freedom — rooted in the security of memory — is what makes bold programming possible.
