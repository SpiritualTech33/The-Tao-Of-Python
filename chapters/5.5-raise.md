# 5.5 raise — Creating and Propagating Errors

## You Can Speak in Exceptions Too

So far, exceptions have come *at* you. Python raises them; you handle them.

But you can also raise exceptions yourself. This is not aggression — it is honesty. When your code encounters a condition it cannot handle, the right response is often to say so loudly, clearly, and specifically.

---

## `raise` — Raising an Exception

```python
def calculate_batting_average(hits: int, at_bats: int) -> float:
    if at_bats < 0:
        raise ValueError(f"at_bats cannot be negative, got {at_bats}")
    if hits < 0:
        raise ValueError(f"hits cannot be negative, got {hits}")
    if hits > at_bats:
        raise ValueError(f"hits ({hits}) cannot exceed at_bats ({at_bats})")

    return hits / at_bats if at_bats > 0 else 0.0
```

This function doesn't silently return wrong answers or quietly produce nonsense. It names the problem and stops — clearly, with useful information.

---

## When to Raise

Raise an exception when:

- **Preconditions are violated** — the caller passed invalid arguments
- **The function cannot fulfill its contract** — it promised to return a value but genuinely cannot
- **A state is invalid** — your system has entered an impossible situation

Do *not* raise exceptions for control flow. Exceptions are not `goto`. They are for genuine error conditions.

```python
# Wrong — using exception for normal flow control
def find_player(players: list, name: str) -> dict:
    for player in players:
        if player["name"] == name:
            raise FoundIt(player)   # No. This is not an error.
    return None

# Right — return a value, use exceptions only for errors
def find_player(players: list, name: str) -> dict | None:
    for player in players:
        if player["name"] == name:
            return player
    return None
```

## Custom Exceptions

You can define your own exception classes. This is how you build a vocabulary of errors specific to your domain:

```python
# Define custom exceptions by inheriting from Exception
class InvalidAverageError(ValueError):
    """Raised when a batting average is outside the valid range [0.0, 1.0]."""
    pass


class PlayerNotFoundError(Exception):
    """Raised when a requested player cannot be found in the roster."""

    def __init__(self, name: str):
        self.name = name
        super().__init__(f"Player '{name}' not found in roster")


# Using custom exceptions
def validate_batting_average(average: float) -> None:
    """Raise InvalidAverageError if average is outside [0.0, 1.0]."""
    if not isinstance(average, (int, float)):
        raise TypeError(f"Expected numeric value, got {type(average).__name__}")
    if average < 0.0 or average > 1.0:
        raise InvalidAverageError(
            f"Batting average must be between 0.0 and 1.0, got {average}"
        )


def get_player(roster: list, name: str) -> dict:
    """Return player dict or raise PlayerNotFoundError."""
    for player in roster:
        if player["name"] == name:
            return player
    raise PlayerNotFoundError(name)
```

Custom exceptions let callers catch *exactly* what they care about — not too broadly, not too narrowly.

## Exception Chaining — `raise ... from`

When you catch one exception and raise another, use `from` to preserve the cause:

```python
def load_player_data(filepath: str) -> dict:
    try:
        with open(filepath) as f:
            raw = f.read()
        return parse_json(raw)
    except FileNotFoundError as e:
        raise PlayerNotFoundError(filepath) from e
```

The `from e` part means: *I am raising a new exception, and the original cause was this other exception.* The traceback will show both, giving you the full chain of causality.

Without `from`, you lose the original context. With it, you preserve the full story.

---

## Re-raising Exceptions

Sometimes you want to do something (log, clean up) and then let the exception continue upward:

```python
def process_file(filepath: str) -> None:
    try:
        data = read_file(filepath)
        process(data)
    except Exception as e:
        log_error(f"Failed to process {filepath}: {e}")
        raise   # Re-raise the same exception, preserving the original traceback
```

Bare `raise` (with no argument) re-raises the current exception exactly as it was. This lets you add side effects (logging) without consuming the exception or changing its type.

## The Hierarchy of Your Own Domain

A well-designed module often defines a base exception and specific ones:

```python
# Base exception for your entire module
class BaseballError(Exception):
    """Base exception for all baseball module errors."""
    pass


class InvalidStatError(BaseballError, ValueError):
    """Raised when a statistical value is invalid."""
    pass


class PlayerNotFoundError(BaseballError):
    """Raised when a player cannot be found."""
    pass


class RosterFullError(BaseballError):
    """Raised when attempting to add a player to a full roster."""
    pass
```

Now callers can choose: catch `BaseballError` to handle anything baseball-related, or catch `PlayerNotFoundError` specifically for just that case.

This is API design. Your exceptions are part of your interface.

---

## The Practice of Honest Failure

There is something clarifying about a function that says, clearly, *"I cannot do this because..."* It does not pretend. It does not return a wrong answer. It does not crash in a cryptic way at some distant point in the code.

It fails at the right place, with the right message, in the right voice.

That is not weakness. That is integrity.
