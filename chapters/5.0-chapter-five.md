# Chapter 5 — When Things Break

> *"The master does not fear the broken vase. He studies the shape of the shards."*

---

## The Question This Chapter Asks

What do you do when your code meets the world — and the world refuses to cooperate?

A file that doesn't exist. A number where a string was expected. A network that goes silent mid-transfer. A user who types `"banana"` when asked for their age.

These are not failures of the programmer. They are **facts of the world.** And a program that cannot face these facts is not finished — it is fragile.

This chapter is about resilience.

---

## Two Ways to Face Error

There is the programmer who writes code and hopes nothing goes wrong.

And there is the programmer who writes code that *knows* things will go wrong — and prepares for it.

The first programmer builds in fair weather. The second builds for the storm.

Python gives you tools to be the second. But the tools are not the lesson. The lesson is **the posture** — the willingness to look at your code and ask: *what could fail here?*

## The Philosophical Foundation

In Zen, there is a koan: *"What is your original face before your parents were born?"*

In programming, there is a quieter version: *"What is your program's behavior before the user arrives?"*

You cannot know. You can only prepare.

An unhandled exception is a program that encountered reality and had no response. A handled exception is a program that looked at reality, named it, and chose what to do next.

The difference is awareness.
