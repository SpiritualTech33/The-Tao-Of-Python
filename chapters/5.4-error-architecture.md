# 5.4 Error Architecture — try, except, else, finally

## The Four Chambers

Python's error handling is built around a four-part structure. Each part has a distinct role. Together they let you respond to any situation with precision.

```python
try:
    # The action you want to attempt
    ...
except SomeException:
    # What to do if that specific exception occurs
    ...
else:
    # What to do if NO exception occurred (success path)
    ...
finally:
    # What to do NO MATTER WHAT (always runs)
    ...
```

You don't need all four every time. Use what the situation requires.

---

## `try` — The Attempt

The `try` block contains the code that *might* fail. Only that code. Nothing else.

```python
# Too broad — hard to know which line raised the error
try:
    data = load_file(path)
    cleaned = clean_data(data)
    result = process(cleaned)
    save_result(result)
except Exception:
    print("Something went wrong")

# Better — narrow the try block
try:
    data = load_file(path)
except FileNotFoundError:
    data = default_data()

cleaned = clean_data(data)
result = process(cleaned)
save_result(result)
```

Keep your `try` blocks small. The smaller the try block, the more precisely you know what failed.

---

## `except` — The Response

The `except` clause catches exceptions. You can be specific or catch multiple types:

```python
# Catch one specific exception
try:
    value = int(user_input)
except ValueError:
    value = 0

# Catch multiple exceptions separately
try:
    result = process(data)
except ValueError as e:
    print(f"Bad value: {e}")
    result = None
except TypeError as e:
    print(f"Wrong type: {e}")
    result = None

# Catch multiple exceptions with same handler
try:
    result = process(data)
except (ValueError, TypeError) as e:
    print(f"Input error: {e}")
    result = None
```

The `as e` binds the exception object to a name so you can inspect it — get the error message, re-raise it, log it.

---

## `else` — The Success Path

`else` runs only if the `try` block completed **without raising any exception.** It is the success path:

```python
def load_and_process(filepath: str) -> dict | None:
    try:
        with open(filepath) as f:
            raw = f.read()
    except FileNotFoundError:
        print(f"File not found: {filepath}")
        return None
    else:
        # This only runs if open() succeeded
        return parse(raw)
```

Why use `else` instead of putting the code after `try`? Because code in `else` is not protected by the `except` clause. If `parse(raw)` raises an error, it will *not* be caught by the `except FileNotFoundError`. This is the right behavior — a parse error is not a file error.

`else` clarifies intent: *this code runs on success, and errors here are different errors.*

---

## `finally` — The Guarantee

`finally` always runs — whether an exception was raised, caught, re-raised, or never occurred:

```python
def read_file(filepath: str) -> str:
    file = open(filepath)
    try:
        return file.read()
    except IOError as e:
        print(f"Read error: {e}")
        return ""
    finally:
        file.close()   # Always closes the file, no matter what
```

`finally` is for **cleanup:** closing files, releasing locks, closing database connections, restoring state. Anything that *must* happen regardless of what went wrong.

In practice, Python's `with` statement handles most cleanup automatically:

```python
# Better — 'with' handles cleanup via context manager
with open(filepath) as file:
    return file.read()
# File automatically closed here, even on exception
```

But `finally` remains essential for resources that don't have a context manager, or for complex cleanup logic.

## Putting It Together

```python
def safe_read_stats(filepath: str) -> dict | None:
    """
    Read player stats from a file.

    Returns parsed dict on success, None on failure.
    Always logs the attempt.
    """
    print(f"Attempting to read: {filepath}")
    result = None

    try:
        with open(filepath) as f:
            raw_data = f.read()
    except FileNotFoundError:
        print(f"File not found: {filepath}")
    except PermissionError:
        print(f"Permission denied: {filepath}")
    else:
        # Only reached if file read succeeded
        result = parse_stats(raw_data)
        print("Stats loaded successfully.")
    finally:
        # Always runs — good for logging, metrics, cleanup
        print(f"Read attempt complete for: {filepath}")

    return result
```

---

## The Golden Rules of Error Handling

1. **Be specific** — catch the exception you expect, not `Exception` or `BaseException`
2. **Keep try blocks small** — narrow scope means precise error attribution
3. **Don't swallow exceptions silently** — at minimum, log them
4. **Use `else` for the success path** — clarifies intent and isolates failure modes
5. **Use `finally` for guaranteed cleanup** — or `with` for managed resources
6. **Never use bare `except:`** — it catches `KeyboardInterrupt` and `SystemExit`, swallowing Ctrl+C

```python
# This is almost never right
try:
    do_something()
except:    # Catches EVERYTHING including SystemExit and KeyboardInterrupt
    pass   # Silently swallows errors — debugging nightmare
```

Silence is not safety. Silence hides the injury until it becomes catastrophic.
