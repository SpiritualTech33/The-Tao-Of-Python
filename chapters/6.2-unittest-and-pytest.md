# 6.2 unittest and pytest — Two Paths

## The Standard Library Path: `unittest`

Python ships with `unittest` — a testing framework built into the standard library. No installation required. Modeled after Java's JUnit.

```python
import unittest

def add(a: float, b: float) -> float:
    return a + b

class TestAdd(unittest.TestCase):

    def test_positive_numbers(self):
        self.assertEqual(add(2, 3), 5)

    def test_negative_numbers(self):
        self.assertEqual(add(-1, -1), -2)

    def test_mixed(self):
        self.assertEqual(add(-1, 1), 0)

    def test_floats(self):
        self.assertAlmostEqual(add(0.1, 0.2), 0.3, places=10)

if __name__ == "__main__":
    unittest.main()
```

Run it:
```bash
python -m unittest test_file.py -v
```

`unittest` is structured, explicit, and requires no extra installation. It is the right choice when you cannot add dependencies.

---

## The Community Path: `pytest`

`pytest` is the dominant testing framework in the Python ecosystem. It is more concise, more powerful, and more Pythonic than `unittest`:

```python
import pytest

def add(a: float, b: float) -> float:
    return a + b

def test_positive_numbers():
    assert add(2, 3) == 5

def test_negative_numbers():
    assert add(-1, -1) == -2

def test_mixed():
    assert add(-1, 1) == 0

def test_floats():
    assert add(0.1, 0.2) == pytest.approx(0.3)
```

Same tests, less ceremony. `pytest` just uses regular functions and regular `assert`.

Run it:
```bash
python -m pytest test_file.py -v
```

## Key Differences

| Feature | `unittest` | `pytest` |
|---------|-----------|---------|
| Setup | Built-in | `pip install pytest` |
| Test structure | Class-based | Functions or classes |
| Assertions | `self.assertEqual(...)` | `assert a == b` |
| Parametrize | Verbose | `@pytest.mark.parametrize` |
| Fixtures | `setUp/tearDown` | `@pytest.fixture` |
| Output | Less readable | Rich, color-coded |
| Plugins | Few | Hundreds |

`pytest` also runs `unittest`-style tests. It is fully compatible. You can use both in the same codebase.

## Parametrize — Testing Many Cases at Once

`pytest`'s `@pytest.mark.parametrize` is one of its most powerful features:

```python
import pytest
from stats import calculate_batting_average

@pytest.mark.parametrize("hits, at_bats, expected", [
    (3, 10, 0.300),
    (0, 5, 0.000),
    (5, 5, 1.000),
    (34, 100, 0.340),
    (0, 0, 0.000),     # Edge case
])
def test_calculate_batting_average(hits, at_bats, expected):
    result = calculate_batting_average(hits, at_bats)
    assert result == pytest.approx(expected, abs=1e-3)
```

This is one test function that actually runs **five times** — once for each tuple of parameters. The output shows each case separately, and each can fail independently.

Without parametrize, you would write five separate test functions or loop inside one. Both approaches obscure either structure or individual failure reporting.

## `pytest.approx` — Floating Point Comparison

Never compare floats with `==`:

```python
# This can fail due to floating point precision
assert 0.1 + 0.2 == 0.3   # False! 0.1 + 0.2 = 0.30000000000000004

# Use pytest.approx
assert 0.1 + 0.2 == pytest.approx(0.3)             # True — within 1e-6
assert 0.1 + 0.2 == pytest.approx(0.3, abs=1e-9)   # Custom tolerance
```

`pytest.approx` checks that two values are close enough, using a configurable tolerance. This is the correct way to test floating point arithmetic.

## Running Tests — Common Commands

```bash
# Run all tests in the current directory (recursively)
python -m pytest

# Run a specific file
python -m pytest test_batting.py

# Run with verbose output (each test named)
python -m pytest -v

# Run and stop on first failure
python -m pytest -x

# Run tests matching a name pattern
python -m pytest -k "average"

# Show print() output from tests
python -m pytest -s

# Run all of the above combined
python -m pytest -v -x -s
```

---

## Test Discovery

pytest automatically discovers tests by looking for:
- Files named `test_*.py` or `*_test.py`
- Functions starting with `test_`
- Classes starting with `Test`
- Methods inside those classes starting with `test_`

No registration required. Name the file correctly, name the functions correctly — pytest finds them.

---

## The Monastery of Tests

Think of `unittest` as the monastery: strict, structured, ancient lineage, everything in its place.

Think of `pytest` as the dojo: open, focused, less ceremony, faster movement.

Both are legitimate paths. The goal of both is the same: to prove that your code behaves as you intend.

In this course, we use `pytest`. But know that `unittest` exists and that reading its tests is a professional skill.
