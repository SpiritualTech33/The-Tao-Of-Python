# 5.1 Error Handling — The Philosophy of Breaking

## Why Errors Are Not the Enemy

An error is not a failure of your program. It is your program being **honest about the world.**

When Python raises a `FileNotFoundError`, it is not breaking. It is telling you the truth: *this file does not exist.* When it raises a `ZeroDivisionError`, it is not malfunctioning. It is respecting mathematics.

The error is the signal. What you do with the signal — that is the craft.

---

## What Is an Exception?

An **exception** is an event that disrupts the normal flow of a program. Python represents exceptions as objects — instances of exception classes that carry information about what went wrong and where.

When an exception is not handled, Python prints a **traceback** and the program stops:

```python
# This program does not handle its errors
number = int("banana")   # Raises ValueError

# Python output:
# Traceback (most recent call last):
#   File "script.py", line 2, in <module>
#     number = int("banana")
# ValueError: invalid literal for int() with base 10: 'banana'
```

The traceback is actually useful. It tells you:
1. **Where** the error occurred (file and line)
2. **What** went wrong (the exception type)
3. **Why** it happened (the error message)

Read your tracebacks. They are maps.

---

## The Anatomy of a Traceback

```
Traceback (most recent call last):      ← Always read bottom-up
  File "script.py", line 10, in main    ← Where you called the function
    result = divide(a, b)
  File "script.py", line 3, in divide   ← Where the error happened
    return a / b
ZeroDivisionError: division by zero     ← The exception type and message
```

Start at the **bottom** — that is the error. Work upward to understand how execution arrived there.

## The Error Hierarchy

Python's exceptions are organized as a tree. Every exception inherits from `BaseException`. Most exceptions you will encounter inherit from `Exception`:

```
BaseException
├── SystemExit              ← Program intentionally exits
├── KeyboardInterrupt       ← User presses Ctrl+C
└── Exception               ← All "regular" errors
    ├── ValueError          ← Right type, wrong value
    ├── TypeError           ← Wrong type entirely
    ├── NameError           ← Name not defined
    ├── IndexError          ← List index out of range
    ├── KeyError            ← Dict key not found
    ├── FileNotFoundError   ← File doesn't exist
    ├── ZeroDivisionError   ← Division by zero
    └── ... (many more)
```

Understanding this hierarchy matters. When you catch `Exception`, you catch *almost everything.* When you catch `ValueError`, you catch only value problems.

**Be specific in your catches.** A net that catches everything catches things you did not mean to catch.

## The Spiritual Dimension

In Buddhism, the First Noble Truth is *dukkha* — often translated as "suffering," but more precisely: *impermanence, instability, the nature of things to break.*

Code runs in the world. The world is impermanent.

A file gets deleted. A network drops. A user enters the unexpected. These are not aberrations. They are **the normal condition of things.**

Error handling is not pessimism. It is clarity about the nature of the world your code inhabits.

---

## Practical Stance

When you write a function, ask:

- What could my caller pass that I don't expect?
- What external resources does this function depend on?
- What happens if those resources are unavailable?
- What should the caller of this function know if something fails?

These questions are not distractions from writing code. They *are* writing code. The function that handles its errors is more honest, more usable, and more professional than the function that silently assumes everything will work.
