# 6.1 Philosophy of Testing — Epistemology of Correctness

## What Does It Mean to Know?

Epistemology is the branch of philosophy concerned with knowledge: what it is, how we get it, and when we are justified in claiming it.

The programmer faces an epistemological crisis every time they write a function: *How do I know this works?*

There are several bad answers:

- "I read the code and it looks right."
- "I ran it once and it returned the right answer."
- "I wrote it carefully."

And one honest answer: *I have written tests that verify its behavior across many cases, and those tests pass.*

---

## The Four Levels of Confidence

```
Level 0 — I wrote it and I trust myself
Level 1 — I ran it manually and it seemed fine
Level 2 — I wrote a test that checks one case
Level 3 — I wrote tests that check many cases, edge cases, and failure modes
```

Most production bugs come from programmers who stopped at Level 0 or 1.

Level 3 is not perfectionism. It is the minimum for code that will be used by others, or by your future self who has forgotten how it works.

---

## Testing as Scientific Method

The scientific method:
1. Observe a phenomenon
2. Form a hypothesis
3. Design an experiment that could **disprove** the hypothesis
4. Run the experiment
5. Accept or revise based on results

Testing:
1. Write a function
2. Hypothesize: *this function should return X when given Y*
3. Write a test that will **fail** if the hypothesis is wrong
4. Run the test
5. Fix or trust based on results

The similarity is not superficial. Both are systematic ways of distinguishing what we *believe* from what we *know.*

---

## The Test as Documentation

A well-written test is documentation that cannot lie.

Comments can become outdated. Docstrings can drift from the actual behavior. READMEs can be written once and never updated.

But a test that passes is always telling the truth — right now, today, with the current code. It describes behavior that is actually present.

```python
@pytest.mark.parametrize("hits, at_bats, expected", [
    (3, 10, 0.3),
    (0, 5, 0.0),
    (5, 5, 1.0),
    (0, 0, 0.0),     # Edge case: no at-bats
])
def test_calculate_batting_average(hits, at_bats, expected):
    assert calculate_batting_average(hits, at_bats) == pytest.approx(expected)
```

This test says more about `calculate_batting_average` than any paragraph of prose could. It shows inputs, shows outputs, and proves the relationship is real.

## TDD — Test-Driven Development

There is a practice in which you write the test *before* the function.

1. Write a test for a function that doesn't exist yet
2. Run the test — it fails (of course — the function doesn't exist)
3. Write the minimum code to make it pass
4. Refactor if needed
5. Repeat

This is called **Test-Driven Development (TDD).** It sounds counterintuitive. It feels strange. But it forces you to think about *what the function should do* before thinking about *how it should work.*

The test becomes a specification. You are not guessing what the function should do. You have written it down, in executable form, before you wrote a single line of implementation.

This changes how you design code. Functions written test-first tend to be smaller, more focused, and easier to test. Because you had to test them before you wrote them.

---

## What Tests Cannot Do

Tests can prove that code works in the cases they cover. They cannot prove that code works in all cases they don't cover.

A passing test suite is not proof of correctness. It is evidence of correctness *in the dimensions you measured.*

This is why writing tests is an act of judgment: choosing which cases matter, which edge cases to cover, which failure modes to anticipate. A test suite is not a complete specification of the universe. It is a specific conversation between you and your code.

Write the conversations that matter.

## The Spiritual Dimension

In Zen Buddhism, there is the practice of *shoshin* — beginner's mind. The expert sees few possibilities. The beginner sees many.

Approaching your own code with beginner's mind means asking: *What if this assumption is wrong? What if the user does the unexpected? What if the system state is not what I imagine?*

Tests are beginner's mind made executable. They question your assumptions. They test your beliefs. They are the part of you that doesn't take your code's word for it.

Cultivate that doubt. It is not weakness. It is the foundation of trustworthy systems.
