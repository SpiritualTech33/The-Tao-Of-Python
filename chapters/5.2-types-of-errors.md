# 5.2 Types of Errors — Naming the Darkness

## Three Kinds of Wrong

Before you can handle errors, you must be able to recognize them. Python errors fall into three categories, and knowing which kind you have changes everything about how you respond.

---

## 1. Syntax Errors — Before the Program Even Runs

A **syntax error** means Python cannot even parse your code. The interpreter refuses to begin.

```python
# SyntaxError — missing colon
def greet(name)
    print(name)

# SyntaxError — invalid assignment
5 = x

# SyntaxError — unmatched parenthesis
result = (1 + 2
```

Python will not run a file with a syntax error. It will tell you the line and, usually, what it expected.

**These are discovered before runtime.** Fix them before anything else. A linter or IDE catches most of these before you even run your code.

---

## 2. Runtime Errors — While the Program Runs

**Runtime errors** (exceptions) occur while the program is executing. The syntax was valid — Python understood your intent. But the *execution* failed:

```python
# ValueError — valid type, invalid value
int("banana")

# TypeError — wrong type
"hello" + 5

# ZeroDivisionError
10 / 0

# NameError — variable doesn't exist
print(undefined_variable)

# IndexError — list position out of range
items = [1, 2, 3]
items[10]

# KeyError — dictionary key missing
player = {"name": "Cosmos"}
player["average"]   # KeyError: 'average'

# AttributeError — object has no such attribute
"hello".fly()

# FileNotFoundError
open("ghost_file.txt")
```

These are the errors you will handle with `try/except`.

---

## 3. Logical Errors — The Invisible Kind

**Logical errors** are the hardest. The code runs. No exception is raised. But the answer is wrong.

```python
def calculate_batting_average(hits: int, at_bats: int) -> float:
    return at_bats / hits   # ← Wrong order. No error raised. Wrong answer.

calculate_batting_average(3, 10)   # Returns 3.33 instead of 0.3
```

Python cannot catch a logical error. Only **tests** can catch logical errors. This is why Chapter 6 exists.

## The Common Exceptions — A Field Guide

| Exception | Cause | Example |
|-----------|-------|---------|
| `ValueError` | Right type, wrong value | `int("abc")` |
| `TypeError` | Wrong type | `"x" + 1` |
| `NameError` | Name not defined | `print(x)` where `x` doesn't exist |
| `IndexError` | Sequence index out of range | `lst[99]` on a 3-item list |
| `KeyError` | Dictionary key not found | `d["missing"]` |
| `AttributeError` | Object has no such attribute | `"str".unknownmethod()` |
| `ZeroDivisionError` | Dividing by zero | `10 / 0` |
| `FileNotFoundError` | File doesn't exist | `open("ghost.txt")` |
| `ImportError` | Module not found | `import nonexistent_module` |
| `StopIteration` | Iterator exhausted | Advancing past end of iterator |
| `RecursionError` | Infinite recursion | Stack depth exceeded |
| `MemoryError` | Out of memory | Processing data too large |

---

## Reading the Hierarchy

Understanding inheritance in exceptions matters. `FileNotFoundError` is a subclass of `OSError`. `ZeroDivisionError` is a subclass of `ArithmeticError`. `ArithmeticError` is a subclass of `Exception`.

```python
# This catches only FileNotFoundError
except FileNotFoundError:
    ...

# This catches FileNotFoundError AND PermissionError AND other OS errors
except OSError:
    ...

# This catches almost everything — usually too broad
except Exception:
    ...
```

**Be as specific as you can.** Catch the exact exception you expect, handle it precisely. Catching `Exception` is a last resort, not a first instinct.

---

## One Philosophical Note

Each exception type is Python's vocabulary for the unexpected. When you see `KeyError`, Python is saying: *you assumed a key existed that did not.* When you see `TypeError`, it is saying: *you assumed a type that was wrong.*

Learning to read these exceptions fluently is learning to understand **your assumptions.** Every exception reveals an assumption you made that reality did not honor.

That is not an insult. That is information.
