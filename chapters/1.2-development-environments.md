# 1.2 Development Environments

## Where Do You Write Python?

Your development environment is your dojo. The place where practice happens. Choose it with intention.

---

## Interactive Mode — The REPL

REPL stands for **Read-Eval-Print Loop.** Open your terminal and type `python3`:

```bash
$ python3
>>> 2 + 2
4
>>> name = "Cosmos"
>>> print(name)
Cosmos
>>> exit()
```

Python reads your input, evaluates it, prints the result, and loops back. **Immediate feedback.** No files, no compilation. Perfect for experimentation and exploration.

The REPL is where you play. Where you ask *"what happens if I do this?"* without consequences.

---

## Script Mode — Files

For real work, you write `.py` files:

```python
# my_first_script.py
name = "Cosmos"
language = "Python"
print(f"Hello. I am {name} and I speak {language}.")
```

Run it from the terminal:

```bash
python3 my_first_script.py
# Hello. I am Cosmos and I speak Python.
```

A script is permanent. It can be saved, shared, version-controlled, run again and again. This is where real code lives.

## VS Code — Your Full Dojo

For serious development, use **Visual Studio Code** with the Python extension.

**Why VS Code?**
- Syntax highlighting — errors become visible before you run the code
- IntelliSense — autocomplete that understands your code
- Integrated terminal — no window switching
- Debugger — pause execution and inspect variables
- Extensions for everything

**Setup:**
1. Download VS Code from `code.visualstudio.com`
2. Install the Python extension by Microsoft
3. Open a folder → create a `.py` file → start writing

## Virtual Environments — Your Isolated Dojo

A virtual environment is an isolated Python installation for your project. Dependencies installed here don't affect anything else on your system.

```bash
# Create
python3.11 -m venv .venv

# Activate (macOS/Linux)
source .venv/bin/activate

# Activate (Windows)
.venv\Scripts\activate

# You'll see (.venv) in your terminal prompt
# Install packages
pip install requests

# Deactivate when done
deactivate
```

**Why does this matter?** Project A needs library version 1.0. Project B needs version 2.0. Without virtual environments, they conflict. With them, each project has its own clean space.

## The Professional Triad

Every serious Python project has three things:

```
my-project/
├── .venv/            ← virtual environment
├── requirements.txt  ← list of dependencies
└── main.py           ← your code
```

```bash
# Save your dependencies
pip freeze > requirements.txt

# Someone else can reproduce your environment exactly
pip install -r requirements.txt
```

This is how professionals share projects. This is how teams collaborate without *"it works on my machine"* problems.
